1.1 — ABAP Orientado a Objetos

🏗️
Classes e Interfaces — Fundamentos
O pilar de todo o desenvolvimento S/4HANA moderno
CORE

Em ABAP clássico programava-se de forma procedural (FORMs, Function Modules). No S/4HANA, tudo é Orientado a Objetos. Classes, interfaces e herança são pré-requisito para qualquer desenvolvimento moderno — incluindo BADIs, RAP e CDS.

Anatomia de uma Classe ABAP
  • CLASS ... DEFINITION — declara a estrutura: atributos, métodos, constantes, tipos, eventos. É o "contrato" da classe.
  • CLASS ... IMPLEMENTATION — implementa os corpos dos métodos declarados na DEFINITION.
  • PUBLIC SECTION — visível para qualquer código externo à classe.
  • PROTECTED SECTION — visível dentro da classe e nas subclasses (herança).
  • PRIVATE SECTION — visível apenas dentro da própria classe. Encapsulamento.
  • CONSTRUCTOR — método especial chamado automaticamente ao criar instância com NEW ou CREATE OBJECT.
  • CLASS-METHODS / CLASS-DATA — membros estáticos, acedidos sem instância via NomeDaClasse=>método.
ABAPClasse completa com constructor, métodos e herança

Polimorfismo — o conceito mais importante de OO

Polimorfismo permite tratar objectos de tipos diferentes de forma uniforme através de uma interface comum. É muito usado nos BADIs e no RAP.

ABAPPolimorfismo via interface
"Pode guardar qualquer veículo que implemente zif_veiculo

APPEND NEW zcl_carro( iv_marca = 'BMW'     iv_ano = 2023 ) TO lt_veiculos.
APPEND NEW zcl_mota( iv_marca  = 'Honda'   iv_ano = 2022 ) TO lt_veiculos.
APPEND NEW zcl_camiao( iv_marca = 'Scania'  iv_ano = 2021 ) TO lt_veiculos.

"Chama o mesmo método em todos — cada um executa o SEU acelerar
LOOP AT lt_veiculos INTO DATA(lo_v).
  lo_v->acelerar( 100 ).
  WRITE / lo_v->get_info( ).
ENDLOOP.

🎯 Perguntas frequentes em entrevista — ABAP OO

  1. "Qual a diferença entre CLASS e INTERFACE?" — Classe pode ter implementação e estado (dados). Interface só declara métodos sem implementar. Uma classe pode implementar N interfaces mas herdar de apenas 1.
  2. "O que é ABSTRACT em ABAP?" — Classe que não pode ser instanciada directamente. Serve como base para herança. Obriga subclasses a implementar métodos ABSTRACT.
  3. "Quando usas CLASS-METHODS vs METHODS?" — CLASS-METHODS (estáticos) não precisam de instância. Úteis para factories, utilidades, constantes. METHODS de instância acedendo a dados da instância específica.
  4. "O que é FINAL em ABAP OO?" — Impede herança (em classe) ou redefinição (em método). Garante que o comportamento não será alterado por subclasses.
⚠️
Tratamento de Excepções — TRY/CATCH
Obrigatório em código profissional — muita gente ignora
CORE

Em ABAP moderno, excepções baseadas em classes substituem os antigos SY-SUBRC. Toda operação que pode falhar deve ser protegida com TRY/CATCH.

ABAPHierarquia de excepções e TRY/CATCH
"Criar excepção customizada (SE24 ou Eclipse)
"Herda de CX_STATIC_CHECK (verificada) ou CX_DYNAMIC_CHECK
CLASS zcx_validation_error DEFINITION
  INHERITING FROM cx_static_check PUBLIC FINAL.
  PUBLIC SECTION.
    DATA: gv_field   TYPE string,
          gv_message TYPE string.
    METHODS constructor
      IMPORTING
        iv_field   TYPE string
        iv_message TYPE string.
ENDCLASS.

"Lançar excepção num método
METHOD validar_cliente.
  IF iv_kunnr IS INITIAL.
    RAISE EXCEPTION NEW zcx_validation_error(
      iv_field   = 'KUNNR'
      iv_message = 'Cliente é obrigatório' ).
  ENDIF.
ENDMETHOD.

"Apanhar excepção no chamador
TRY.
    lo_service->validar_cliente( iv_kunnr = lv_kunnr ).
    lo_service->gravar( ).
    MESSAGE 'Gravado com sucesso' TYPE 'S'.

  CATCH zcx_validation_error INTO DATA(lx_val).
    MESSAGE lx_val->gv_message TYPE 'E'.

  CATCH cx_root INTO DATA(lx_root).
    "Apanha qualquer excepção não prevista
    MESSAGE lx_root->get_text( ) TYPE 'E'.
ENDTRY.
⚠️
Tipos de excepção: CX_STATIC_CHECK obriga o chamador a declarar RAISING ou capturar com TRY/CATCH (verificado em tempo de compilação). CX_DYNAMIC_CHECK é verificado em runtime. CX_NO_CHECK propaga automaticamente sem necessidade de declarar.

TYPES: BEGIN OF ty_material,
         matnr TYPE matnr,

🎯 Pergunta clássica: "Quando usas HASHED vs SORTED?"

  1. HASHED — quando só precisas de lookup por chave completa e única. Ex: tabela de materiais usada dentro de um LOOP para enriquecer dados. Não podes iterar por índice.
  2. SORTED — quando precisas de pesquisa por chave parcial (prefixo), por intervalo, ou quando precisas de iterar ordenado. Ex: tabela de posições de uma ordem ordenada por item.
  3. STANDARD — para listas simples onde a ordem de inserção importa, ou quando vais usar SORT + BINARY SEARCH manualmente.

1.3 — Dicionário de Dados ABAP (DDIC)

🗄️
Tabelas, Estruturas, Tipos, Domínios e Elements
A camada de metadata que governa todos os dados SAP
CORE

O DDIC (Data Dictionary) é o repositório central de definições de dados no SAP. Qualquer tabela, campo, tipo ou estrutura passa pelo DDIC antes de existir no banco de dados.

Hierarquia de definição
1
Domínio (Domain)
Define o tipo técnico de dados: CHAR(18), INT4, DATS, CURR. Também define valores fixos (ex: STATUS com valores 'A','I','B'). Ferramenta: SE11 → Domínio.
2
Elemento de Dados (Data Element)
Dá semântica ao campo: rótulos curto/médio/longo, documentação, referência ao domínio. Ex: MATNR usa domínio MATNR mas tem labels "Material", "Nº de material", etc.
3
Estrutura (Structure)
Agrupamento de campos sem tabela física no banco. Usada em parâmetros de Function Modules, BAPIs, e como tipo de work area em programas.
4
Tabela Transparente
Existe 1:1 no banco de dados como tabela real. Tem chave primária, campos, atributos técnicos e índices. Ex: MARA, VBAK, BKPF.
5
View DDIC
JOIN de tabelas definido no DDIC. Database View (executado no DB), Help View (para search help), Maintenance View (para SM30).

1.5 — Debugging Profissional

🔍
Técnicas Avançadas de Depuração SAP
Habilidade que separa júnior de sénior mais rapidamente que qualquer outra
CORE

Debugar bem em SAP é uma arte. Não é só "colocar um breakpoint e ver o que acontece" — é saber exactamente onde parar, o que procurar e como modificar o estado em runtime para testar cenários.

Tipos de breakpoint e quando usar cada um
TipoComo activarQuando usar
EstáticoBREAK-POINT. no códigoDesenvolvimento local. NUNCA em produção — para toda a sessão.
Dinâmico (sessão)Duplo clique na margem do editorPara a tua sessão apenas. Não afecta outros utilizadores.
Externo (user)Utilities → Breakpoints → Set/Delete ExternalPara debugar jobs em background, IDocs, RFC chamados por sistemas externos.
WatchpointNo debugger: Watchpoints → CreatePara automáticamente quando uma variável muda de valor — muito útil para encontrar onde um dado é alterado.
/h no command fieldEscreves /h e ENTER em qualquer transacçãoPara entrar no modo debug a partir do próximo passo.
Fluxo de debugging eficiente
1
Reproduzir o problema
Antes de debugar, entende exactamente os dados de entrada e o resultado esperado vs actual. Documentar o cenário de reprodução.
2
Identificar o ponto de entrada
Qual transacção, qual user-exit/BADI, qual job. Colocar o breakpoint no ponto mais próximo do problema, não no início do programa.
3
Navegar com inteligência
F5 (step into), F6 (step over), F7 (step out), F8 (continue). Usar F6 para pular Function Modules que não são o problema.
4
Analisar variáveis e tabelas
Duplo clique numa variável para ver o valor. Clicar no ícone de tabela para ver tabelas internas completas. Verificar SY-SUBRC após cada operação crítica.
5
Modificar valores em runtime
No painel de variáveis, clica numa variável e altera o valor para testar diferentes cenários sem ter de reiniciar. Poderoso para testar edge cases.
💡
Dica pro: Para debugar um IDoc inbound, usa a WE19 para reprocessar o IDoc com os dados originais e depois activa breakpoint externo para o teu user. O sistema vai parar exactamente no teu código quando o IDoc for reprocessado.
ℹ️
ST22 — Short Dumps: Quando um programa termina abruptamente, o SAP gera um dump no ST22. Sempre analisa: o tipo de excepção, a linha de código que causou, e o stack trace completo. É frequentemente mais rápido que debugar para encontrar a causa.

1.6 — Performance Tuning

⚡
Regras Completas de Performance ABAP
Código lento em produção com milhões de registos é um problema crítico
ENTREVISTA

Performance em ABAP é 90% sobre como interages com o banco de dados e 10% sobre lógica ABAP. As regras abaixo cobrem os problemas mais comuns em código de produção.

Regra #1 — Nunca SELECT dentro de LOOP
ABAP❌ Errado vs ✅ Correcto
"❌ ERRADO — N chamadas ao banco de dados (N = linhas da tabela)
LOOP AT lt_vbap INTO DATA(ls_vbap).
  SELECT SINGLE maktx FROM makt
    INTO ls_vbap-maktx
    WHERE matnr = ls_vbap-matnr AND spras = 'PT'.
ENDLOOP.

"✅ CORRECTO — 2 chamadas ao banco independente do tamanho
"1. Busca todas as posições
SELECT vbeln posnr matnr
  FROM vbap INTO TABLE @lt_vbap
  WHERE vbeln IN so_vbeln.

"2. Busca descrições com FOR ALL ENTRIES
IF lt_vbap IS NOT INITIAL.  "CRÍTICO — sem este check traz TUDO da MAKT!
  SELECT matnr maktx FROM makt
    INTO TABLE @lt_makt
    FOR ALL ENTRIES IN @lt_vbap
    WHERE matnr = @lt_vbap-matnr AND spras = 'PT'.
ENDIF.

"3. Constrói HASHED TABLE para lookup O(1)
DATA(lt_makt_h) = VALUE ty_t_makt_h( FOR ls IN lt_makt ( ls ) ).

"4. Enriquece sem SELECT adicional
LOOP AT lt_vbap ASSIGNING FIELD-SYMBOL(<ls>).
  READ TABLE lt_makt_h INTO DATA(ls_makt)
    WITH TABLE KEY matnr = <ls>-matnr.
  IF sy-subrc = 0.
    <ls>-maktx = ls_makt-maktx.
  ENDIF.
ENDLOOP.
Regra #2 — SELECT apenas os campos necessários
ABAPSELECT eficiente
"❌ Traz 80 campos quando só precisas de 3
SELECT * FROM mara INTO TABLE @lt_mara.

"✅ Só os campos necessários
SELECT matnr mtart mbrsh
  FROM mara
  INTO TABLE @DATA(lt_mara)  "Inline declaration
  WHERE mtart = 'FERT'
    AND mbrsh = 'M'.

"✅ Aggregation no banco — não traz dados para somar em ABAP
SELECT kunnr, SUM( netwr ) AS total_vendas
  FROM vbak
  INTO TABLE @DATA(lt_total)
  WHERE erdat >= lv_data_inicio
  GROUP BY kunnr.
Ferramentas de análise de performance
  • SE30 — ABAP Runtime Analysis — mede tempo de execução por instrução. Identifica bottlenecks precisamente.
  • ST05 — SQL Trace — regista cada chamada ao banco de dados com tempo de execução. Para encontrar SELECT dentro de LOOP e queries sem índice.
  • SAT — ABAP Trace — versão moderna do SE30 com melhor visualização.
  • SM50/SM66 — monitora processos de trabalho em tempo real para ver o que está a consumir recursos.

🔗
IDocs — Arquitectura e Processamento Completo
O formato standard SAP para integração assíncrona entre sistemas
RICEF

Um IDoc (Intermediate Document) é uma estrutura de dados hierárquica com um registo de controlo (EDIDC), um registo de dados (EDID4) e opcionalmente registos de status (EDIDS). É o formato SAP para EDI e integração assíncrona.

Componentes de um IDoc
  • Tipo de IDoc (IDOCTYPE) — define a estrutura hierárquica de segmentos. Ex: ORDERS05, INVOIC02, DESADV01.
  • Tipo de Mensagem (MESTYP) — define o propósito de negócio. Ex: ORDERS (pedido de compra), INVOIC (fatura), SHPMNT (expedição).
  • Segmentos — linhas de dados dentro do IDoc. E1EDKA1 (dados de parceiro), E1EDP01 (posições), etc.
  • Partner Profile (WE20) — configuração de parceiro: quem envia/recebe, qual processamento, qual message type.
  • Port (WE21) — o canal de comunicação: ficheiro, RFC, HTTP.
  • ALE/EDI — ALE para SAP-SAP, EDI para SAP-sistemas externos.

RFC — Remote Function Call
  • sRFC (síncrono) — chama e espera resposta. Usado quando o resultado é necessário imediatamente.
  • aRFC (assíncrono) — chama sem esperar. O programa continua. Resultado ignorado ou tratado depois.

  • qRFC (queued) — como tRFC mas com garantia de ordem de execução dentro de uma fila.

2.3 — C: Conversions (Migração de Dados)

🔄
Programa de Migração Profissional com Log e Modo Teste
A missão mais crítica num projecto de implementação SAP
RICEF

Migrations são irreversíveis em produção. Um erro de migração pode corromper dados de clientes, stock ou contabilidade. Por isso, todo o programa de migração profissional tem: modo de simulação, log detalhado por registo, rollback em caso de erro, e processamento em pacotes.

1
Extrair dados do sistema legado
Normalmente via ficheiro Excel/CSV ou tabelas de staging. Validar qualidade dos dados antes de começar.
2
Carregar para tabela de staging Z
Cria uma tabela Z no SAP com os dados brutos do legado. Permite reprocessar sem reimportar o ficheiro.
3
Programa de validação
Verifica duplicados, campos obrigatórios, referências inválidas. Gera relatório de erros ANTES de criar documentos.
4
Programa de carga com BAPI
Cria documentos SAP via BAPI. Sempre com modo de simulação (sem COMMIT), log por registo, e pacotes de 100-500 registos.
5
Reconciliação
Compara totais do legado vs SAP após migração. Relatório de rastreabilidade (ID legado → nº documento SAP).
ABAPPadrão completo de programa de migração
REPORT zmm_migra_materiais.

PARAMETERS:
  p_test  AS CHECKBOX DEFAULT 'X',  "X = simulação
  p_pack  TYPE i DEFAULT 200.          "Tamanho do pacote

TYPES: BEGIN OF ty_log,
         id_legado TYPE char20,
         matnr     TYPE matnr,
         status    TYPE char1,  "S=sucesso, E=erro
         mensagem  TYPE string,
       END OF ty_log.

DATA: gt_log   TYPE TABLE OF ty_log,
      gv_ok    TYPE i,
      gv_erro  TYPE i.

START-OF-SELECTION.
  "Lê dados da staging table
  SELECT * FROM zmm_stag_mat
    INTO TABLE @DATA(lt_staging)
    WHERE status = 'P'.  "Pendentes

  "Processa em pacotes
  DATA(lv_total) = lines( lt_staging ).
  DATA(lv_inicio) = 1.

  DO.
    DATA(lv_fim) = MIN( lv_inicio + p_pack - 1, lv_total ).

    LOOP AT lt_staging INTO DATA(ls_stag)
      FROM lv_inicio TO lv_fim.

      "Mapeia para estrutura BAPI
      DATA(ls_bapi) = VALUE bapimathead(
        matnr = ls_stag-matnr_legado
        mtart = ls_stag-mtart ).

      "Chama BAPI
      CALL FUNCTION 'BAPI_MATERIAL_SAVEDATA'
        EXPORTING headdata = ls_bapi
        TABLES    return   = DATA(lt_return).

      "Avalia resultado
      DATA(lv_erro) = VALUE #( lt_return[
        KEY type = 'E' OPTIONAL ] ).

      IF lv_erro-type = 'E'.
        APPEND VALUE ty_log(
          id_legado = ls_stag-id_legado
          status    = 'E'
          mensagem  = lv_erro-message ) TO gt_log.
        gv_erro += 1.
        CALL FUNCTION 'BAPI_TRANSACTION_ROLLBACK'.
      ELSE.
        APPEND VALUE ty_log(
          id_legado = ls_stag-id_legado
          status    = 'S'
          mensagem  = 'Migrado com sucesso' ) TO gt_log.
        gv_ok += 1.
        IF p_test IS INITIAL.  "Só faz COMMIT em modo real
          CALL FUNCTION 'BAPI_TRANSACTION_COMMIT'
            EXPORTING wait = 'X'.
        ENDIF.
      ENDIF.
    ENDLOOP.

    lv_inicio = lv_fim + 1.
    IF lv_inicio > lv_total. EXIT. ENDIF.
  ENDDO.

  "Exibe relatório do log via ALV
  WRITE: / |Sucesso: { gv_ok } | Erro: { gv_erro }|.

🎯 Perguntas de entrevista — Enhancements

  1. "Qual a diferença entre User-Exit e BADI?" — User-Exit: FORM procedural, apenas uma implementação. BADI: orientada a objectos, múltiplas implementações possíveis, mais flexível e rastreável.
  2. "Como sabes se existe uma BADI para um processo?" — SE18 por área funcional, activar Enhancement Spot mode na transacção, ou pesquisar na SAP help por "user-exits" do processo.
  3. "O que é filtro numa BADI?" — Mecanismo que permite múltiplas implementações activas, onde cada uma só actua quando determinado critério de filtro é verdadeiro (ex: só para uma sociedade específica).
  4. "O que é um Implicit Enhancement Point?" — Ponto no código standard onde podes adicionar código sem modificar o standard. O SAP define estes pontos e podes vê-los com Edit → Enhancement Operations no Editor.

Define a estrutura de dados do BO. Usa tabela transparente como fonte. Anotação @Semantics.systemDateTime.createdAt para campos de auditoria.
2
Behavior Definition (BDEF)
Declara as operações possíveis: create, update, delete, actions, validations, determinations. Também define campos obrigatórios e de leitura.
3
Behavior Implementation (classe ABAP)
Implementa as validations, determinations e actions. A classe herda de cl_abap_behavior_handler.
4
Projection View e BDEF de Projecção
Camada de consumo com annotations UI. Restringe e adapta o BO para o contexto específico da aplicação.
5
Service Definition e Service Binding
Expõe o BO via OData v2 ou v4. O Service Binding activa o serviço e liga ao protocolo. Também suporta UI5/Fiori e API.

define behavior for Z_I_SalesOrderAO
  alias SalesOrder
  persistent table zsalesorder_ao
  lock master
  authorization master ( instance )
  etag master LastChangedAt

  create;
  update;
  delete;

  field ( readonly ) CreationDate, LastChangedAt, CreatedBy;
  field ( mandatory ) SoldToParty, SalesOrganization;

  "Validações executadas antes de gravar
  validation validateCustomer on save
    { create; field SoldToParty; }

  validation validateDates on save
    { create; update; field RequestedDeliveryDate; }

  "Determination — calcula valores automaticamente
  determination setDefaults on modify
    { create; }

  determination calculateTotals on modify
    { field NetAmount, Tax; }

    result [1] $self;

  action cancelOrder
    parameter ZS_CancelReason
    result [1] $self;

  "Composição — posições pertencem ao cabeçalho

  METHOD validateCustomer.
    "Lê dados dos keys fornecidos pelo framework RAP
    READ ENTITIES OF Z_I_SalesOrderAO
      ENTITY SalesOrder
      FIELDS ( SoldToParty )
      WITH CORRESPONDING #( keys )
      RESULT DATA(lt_orders)
      FAILED DATA(lt_failed).

    LOOP AT lt_orders INTO DATA(ls_order).
      "Verifica se cliente existe
      SELECT SINGLE kunnr FROM kna1
        INTO @DATA(lv_kunnr)
        WHERE kunnr = @ls_order-SoldToParty.

      IF sy-subrc <> 0.
        "Adiciona mensagem de erro ao framework
        APPEND VALUE #(
          %tky        = ls_order-%tky
          %state_area = 'VALIDATE_CUSTOMER'
          %msg        = NEW zcx_sales_msg( textid   = zcx_sales_msg=>invalid_customer
                                               customer = ls_order-SoldToParty )
          %element-SoldToParty = 'X' ) TO reported-SalesOrder.
        APPEND VALUE #( %tky = ls_order-%tky ) TO failed-SalesOrder.
      ENDIF.
    ENDLOOP.
  ENDMETHOD.

  METHOD confirmOrder.
    "Altera status para Confirmado
    MODIFY ENTITIES OF Z_I_SalesOrderAO
      ENTITY SalesOrder
      UPDATE FIELDS ( OverallStatus )
      WITH VALUE #( FOR key IN keys (
        %tky          = key-%tky
        OverallStatus = 'B' ) )  "B = Confirmado
      REPORTED DATA(lt_rep)
      FAILED DATA(lt_fail).
  ENDMETHOD.

  METHOD setDefaults.
    MODIFY ENTITIES OF Z_I_SalesOrderAO
      ENTITY SalesOrder
      UPDATE FIELDS ( OverallStatus CreationDate )
      WITH VALUE #( FOR key IN keys (
        %tky          = key-%tky
        OverallStatus = 'A'           "Aberto
        CreationDate  = cl_abap_context_info=>get_system_date( ) ) ).
  ENDMETHOD.

4.1 — Scrum em Projectos SAP

🏃
Como Scrum funciona na prática em SAP
Do requisito à entrega — o ciclo completo de desenvolvimento
EQUIPA
Fluxo de um Sprint SAP
1
Sprint Planning
O analista funcional apresenta as User Stories. O ABAPer lê as especificações e estima em Story Points ou horas. Comprometem-se com o que cabe no Sprint (2 semanas).
2
Receber a Especificação Técnica
O funcional entrega a "Functional Spec" com: objectivo de negócio, dados de entrada/saída, regras, casos de excepção, transacção afectada. O ABAPer transforma isto numa "Technical Spec".
3
Desenvolvimento em DEV
Desenvolves no ambiente de Desenvolvimento. Testes unitários (ABAP Unit). Pedido de transporte criado automaticamente.
4
Code Review
Outro ABAPer ou o lead técnico revê o código. Verifica: performance, nomenclatura, tratamento de erros, lógica. Pull Request (se usam Git/gCTS).
5
Testes em QAS
Transporta para Quality (QAS). O funcional e o key-user testam. Bugs encontrados voltam para DEV.
6
Go-Live em PRD
Transporte para Produção após aprovação. Suporte de nível 3 nas primeiras semanas para problemas que só aparecem em produção com dados reais.
Documentação técnica — o que escrever
  • Objectivo — o que o desenvolvimento faz em 2-3 frases.
  • Dados de entrada — parâmetros, tabelas de origem, selection screen.
  • Lógica de processamento — fluxo passo a passo, decisões, excepções.
  • Dados de saída — tabelas actualizadas, documentos criados, output ALV/form.
  • Objectos transportados — lista de todos os objectos ABAP criados/alterados com descrição.
  • Transacções afectadas — onde o utilizador vai interagir com o resultado.
  • Guia de teste — passos exactos para o funcional testar, incluindo dados de teste.
